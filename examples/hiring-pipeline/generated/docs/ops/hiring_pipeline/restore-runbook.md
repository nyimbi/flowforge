## Purpose

This is the **disaster-recovery (DR) tabletop runbook** for the
hiring-pipeline bundle. It walks the operator from a cold backup
through to a verified, audit-chain-intact restore on a scratch
database. It is paired with `make restore-drill`, which executes the
same procedure end-to-end against a testcontainers Postgres and
asserts every audit chain re-verifies.

The runbook is derivable: every entity table the bundle creates plus
the optional per-JTBD idempotency-keys table (item 6) plus every audit
topic the bundle emits is declared in the JTBD bundle. If any of those
shape characteristics change, regenerate this file rather than editing
it.

## Required `pg_dump` flags

Two dumps are taken — one schema-only, one data-only — so the restore
step can rebuild the schema cleanly before reloading data without
fighting transactional constraints from a mixed dump.

### Schema dump

```bash
pg_dump \
	--schema-only \
	--no-owner \
	--no-privileges \
	--format=plain \
	--file=hiring_pipeline.schema.sql \
	"$DATABASE_URL"
```

* `--schema-only` — DDL only, no rows. The data-only dump below carries
  the rows.
* `--no-owner` — strip the original owner so the restore target can use
  any role. Standard for portable backups.
* `--no-privileges` — strip GRANT/REVOKE so the restore target can
  re-grant under its own RBAC policy.
* `--format=plain` — readable SQL output. Switch to `--format=custom`
  for a binary dump that supports `pg_restore --jobs=N` parallelism on
  larger datasets.

### Data dump

```bash
pg_dump \
	--data-only \
	--no-owner \
	--disable-triggers \
	--format=plain \
	--file=hiring_pipeline.data.sql \
	--table=complete_hire \
	--table=complete_hire_idempotency_keys \
	--table=conduct_interview \
	--table=conduct_interview_idempotency_keys \
	--table=extend_offer \
	--table=extend_offer_idempotency_keys \
	--table=screen_candidate \
	--table=screen_candidate_idempotency_keys \
	--table=source_candidate \
	--table=source_candidate_idempotency_keys \
	"$DATABASE_URL"
```

* `--data-only` — rows only; no DDL. The schema dump carries the DDL.
* `--disable-triggers` — required for `--data-only` restores so audit
  triggers don't fire during the bulk INSERT.
* `--table=…` — explicit per-table list. The implicit `pg_dump` mode
  picks up host-app tables this runbook doesn't own; restoring those
  is the host's responsibility, not this bundle's.

## Restore procedure

### 1. Verify the dump artefacts

```bash
test -f hiring_pipeline.schema.sql || { echo "missing schema dump"; exit 1; }
test -f hiring_pipeline.data.sql   || { echo "missing data dump"; exit 1; }
sha256sum hiring_pipeline.schema.sql hiring_pipeline.data.sql
```

Record the checksums in your DR ticket. The restore-drill target hashes
the dumps so a future drill can detect drift in the dump producer.

### 2. Provision the scratch database

```bash
createdb -h "$RESTORE_PGHOST" -U "$RESTORE_PGUSER" "$RESTORE_DBNAME"
```

The host's `tenants` table must already exist (or be restored from the
host's own dump) before this bundle's tables load — see *Tables in
restore order* above.

### 3. Apply the schema

```bash
psql "$RESTORE_DATABASE_URL" --single-transaction --file=hiring_pipeline.schema.sql
```

Wraps the entire schema apply in one transaction so a failure rolls
back rather than leaving a half-built schema.

### 4. Run flowforge migrations forward

```bash
cd backend
alembic upgrade head
```

The migrations land idempotently on top of a `--schema-only` restore so
new framework revisions added since the dump was taken come along for
the ride. Revision ids are deterministic
(`sha256(package + jtbd_id)[:12]`) so a downgrade-then-upgrade cycle is
reversible — see `flowforge pre-upgrade-check --alembic-chain`.

### 5. Load the data

```bash
psql "$RESTORE_DATABASE_URL" --single-transaction --file=hiring_pipeline.data.sql
```

`--single-transaction` again — a partial data restore is worse than a
clean failure.

### 6. Re-verify audit chains

For **every** tenant present in the dump, re-verify the audit hash
chain. A successful verify means: (a) no rows missing in the chain, (b)
each row's `prev_hash` matches the prior row's `hash`, (c) the head row
matches the recorded chain-head pointer. A failure here means the dump
was tampered with or the restore corrupted ordering.

```bash
# Enumerate tenants present in the dump.
TENANTS=$(psql "$RESTORE_DATABASE_URL" -At -c \
	"SELECT DISTINCT tenant_id FROM audit_log WHERE tenant_id IS NOT NULL ORDER BY tenant_id;")

# Verify each tenant's audit chain.
for t in $TENANTS; do
	flowforge audit verify --tenant "$t" || { echo "audit chain broken for tenant=$t"; exit 1; }
done
```

The bundle's audit topic catalogue (used by the verifier to know which
event types to expect):

* `complete_hire.approved`
* `complete_hire.candidate_withdrawal_rejected`
* `complete_hire.submitted`
* `conduct_interview.approved`
* `conduct_interview.no_show`
* `conduct_interview.submitted`
* `extend_offer.approved`
* `extend_offer.counter_offer`
* `extend_offer.escalated`
* `extend_offer.offer_expired_rejected`
* `extend_offer.submitted`
* `screen_candidate.approved`

* `source_candidate.approved`

### 7. Smoke test

Boot the FastAPI app against the restored database and fire one
canonical event per JTBD to prove the wires:

```bash
cd backend
uv run uvicorn hiring_pipeline.app:app --port 8080 &
SERVER_PID=$!
sleep 2

# Smoke: Complete hire and onboard (complete_hire)
curl -fsS -X POST "http://localhost:8080/complete-hire/events" \
	-H "Content-Type: application/json" \
	-H "X-Tenant-Id: $SMOKE_TENANT_ID" \
	-d '{"event":"submit","payload":{}}' \
	|| { kill $SERVER_PID; echo "smoke failed for complete_hire"; exit 1; }
# Smoke: Conduct interview loop (conduct_interview)
curl -fsS -X POST "http://localhost:8080/conduct-interview/events" \
	-H "Content-Type: application/json" \
	-H "X-Tenant-Id: $SMOKE_TENANT_ID" \
	-d '{"event":"submit","payload":{}}' \
	|| { kill $SERVER_PID; echo "smoke failed for conduct_interview"; exit 1; }
# Smoke: Extend an offer (extend_offer)
curl -fsS -X POST "http://localhost:8080/extend-offer/events" \
	-H "Content-Type: application/json" \
	-H "X-Tenant-Id: $SMOKE_TENANT_ID" \
	-d '{"event":"submit","payload":{}}' \
	|| { kill $SERVER_PID; echo "smoke failed for extend_offer"; exit 1; }
# Smoke: Screen a candidate (screen_candidate)
curl -fsS -X POST "http://localhost:8080/screen-candidate/events" \
	-H "Content-Type: application/json" \
	-H "X-Tenant-Id: $SMOKE_TENANT_ID" \
	-d '{"event":"submit","payload":{}}' \
	|| { kill $SERVER_PID; echo "smoke failed for screen_candidate"; exit 1; }
# Smoke: Source a candidate (source_candidate)
curl -fsS -X POST "http://localhost:8080/source-candidate/events" \
	-H "Content-Type: application/json" \
	-H "X-Tenant-Id: $SMOKE_TENANT_ID" \
	-d '{"event":"submit","payload":{}}' \
	|| { kill $SERVER_PID; echo "smoke failed for source_candidate"; exit 1; }

kill $SERVER_PID
```

A 4xx is acceptable for the smoke step (validation errors, missing
payload fields, etc.) — what matters is that the server boots and the
router resolves. A 5xx means the restore itself is suspect.

## Drill cadence

The DR tabletop should run **monthly** at minimum. The
`make restore-drill` target runs the same procedure against a
testcontainers Postgres so the drill can be exercised on every PR
without needing a real backup artefact.

```bash
make restore-drill
```

A green run asserts:

* The schema dump applies cleanly to a fresh database.
* The data dump loads without FK violations.
* `alembic upgrade head` is idempotent against the restored schema.
* Every audit chain re-verifies (no orphan rows, hash chain intact).
* The smoke event fires per JTBD.

A red run **must not** be ignored. Failure modes by symptom:

| Symptom | Likely cause | Action |
|---|---|---|
| `pg_restore: schema mismatch` | Dump was taken from a newer schema than the restore target | Run `alembic upgrade head` *before* loading the data dump. |
| `psql: foreign key violation` | Host `tenants` rows missing | Restore the host's dump first, then this bundle's. |
| `flowforge audit verify` fails | Hash chain broken in the dump | Treat the dump as untrusted; investigate the source database. |
| Smoke step returns 5xx | Restored schema is missing migrations the framework now expects | `alembic upgrade head` then re-run smoke. |

target's spans are captured under the `flowforge.restore_drill.*`
