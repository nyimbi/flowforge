## What this is

A shell-only inbound/outbound email adapter for the `insurance-claim-demo`
bundle. The generator emits:

* `inbound.py` — RFC-5322 parser ABC and the reply-to-fire-event
  workflow.
* `templates.py` — outbound notification email templates keyed by
  audit topic.
* `router.py` — reply-subject grammar walker that maps inbound
  messages onto a POST against `/<url-segment>/events`.

## What this is *not*

The generator emits the **routing skeleton only**. Hosts wire the
concrete IMAP / SMTP / SES / SendGrid bridge themselves:

* IMAP credentials and inbox polling cadence;
* SMTP / SES / SendGrid outbound transport;
* DKIM / SPF / DMARC verification (defence in depth — never trust the
  reply-subject token alone);
* identity bridging from the `From:` header to the bundle's tenant +
  user model.

That intentional cut keeps the bundle deterministic and lets each host
pick its own email runtime.

## Operation source

Every reply-subject route is pinned to the same `openapi.yaml` the W1
generator emits at the bundle root (`../../openapi.yaml` relative
to this package). The adapter does **not** re-derive operations from
the bundle.

## Reply-subject grammar

Outbound notifications carry a subject suffix of the form::

	[<jtbd>:<event>:<instance_id>]

Reply-to-fire dispatches an event by parsing the suffix and POSTing to
the matching backend route. The closed catalog of `<jtbd>:<event>`
tokens lives in `inbound.py`:

* `claim_intake:approve` — File an insurance claim (FNOL) (approve)
* `claim_intake:compensate` — File an insurance claim (FNOL) (compensate)
* `claim_intake:escalate` — File an insurance claim (FNOL) (escalate)
* `claim_intake:reject` — File an insurance claim (FNOL) (reject)
* `claim_intake:submit` — File an insurance claim (FNOL) (submit)

The reply body becomes the JSON `payload`; the parser accepts a simple
`key: value` line grammar so manual replies stay readable.

## Outbound templates

The bundle's notification adapter posts emails through the templates
in `templates.py`. One template per audit topic the bundle aggregates:

* `claim_intake.approved`
* `claim_intake.escalated`
* `claim_intake.lapsed_rejected`
* `claim_intake.large_loss`
* `claim_intake.submitted`

## Wiring

Hosts subclass `EmailAdapter` from `router.py` and implement two
hooks:

* `verify_envelope(message)` — DKIM/SPF/DMARC and From-header
  validation.
* `resolve_principal(message)` — bridge the sender's email to the
  bundle's tenant + user model.

Then call `dispatch(message)` from your IMAP poller. The router takes
care of subject parsing and POSTing to the right backend endpoint.
