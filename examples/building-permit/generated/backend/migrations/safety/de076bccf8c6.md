runtime data-loss or write-amplification risk because no rows pre-date
the migration. The static analyzer classifies the headline severity as
INFO and surfaces the forward-looking risk register below.

### NOT NULL columns (1)

The following columns are emitted with `nullable=False` against an empty
table — safe at first-create. **Never** add a NOT NULL column to an
existing populated table without first landing a backfill.

- `decision` (enum → `String(64)`)

**Suggested rewrite for future NOT NULL adds.** Split the change into
three migrations:

1. add the column nullable;
2. backfill in a separate, online migration;
3. add NOT NULL with `op.alter_column(..., nullable=False)` after the
   backfill is verified.

### Indexes

  created inside the same `op.create_table` transaction. Safe for
  first-create only.

**Suggested rewrite for future index adds against populated tables.**
Postgres hosts must use `op.create_index(..., postgresql_concurrently=True)`
in a migration that opts out of the implicit transaction wrapper
(`# revision is not transactional` plus `with op.get_context().autocommit_block():`).
The `db_migration` template emits the non-CONCURRENTLY form on the
assumption the table is empty at creation time; once any data exists
in production, switch to the CONCURRENTLY form.

## Forward-looking risk register

| Operation | Severity if applied to populated table | Suggested rewrite |
|---|---|---|
| Drop column | HIGH | Land a deprecation window first; coordinate with consumers; flip to NULL+default before drop. |
| Narrow column type (e.g. `String(512)` → `String(64)`) | HIGH | Validate every existing row fits the new bound; backfill or reject mismatches; then narrow. |
| Add NOT NULL column | HIGH | Three-step add-nullable / backfill / alter-NOT-NULL. |
| `CREATE INDEX` without `CONCURRENTLY` | HIGH (Postgres) | Use `postgresql_concurrently=True` and a non-transactional migration. |
| Multi-head merge | CRITICAL | Resolve with `alembic merge heads`; never deploy with two heads. |
