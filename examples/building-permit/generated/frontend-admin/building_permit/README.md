# building-permit — admin console

## What this is

A tenant-scoped operator console for the `building-permit` bundle.
This is a **separate** React app from the customer-facing `frontend/`
tree — operators get a different UX and a different threat model, so
the two ship as independent SPAs behind their own ingress + auth
proxies.

The console covers six panels:

1. **Instance browser** — list entities, filter by JTBD/state/tenant.
2. **Audit-log viewer** — calls `AuditSink.verify_chain()` and renders
   the hash-chain integrity status per topic.
3. **Saga compensation panel** — pending compensations + manual
   trigger gated by `admin.<jtbd>.compensate`. Until the OTel work
   (item 12) lands the per-saga span list shows "no spans recorded";
   that's expected.
4. **Permission-grant history** — who granted what to whom, when
   (sourced from `AccessGrantPort`).
5. **Deferred outbox queue** — envelopes pending dispatch + retry
   status (sourced from `OutboxRegistry`).
6. **RLS elevation log** — records of `elevation_scope()` calls
   (sourced from the audit chain's `rls.elevate` topic).

## Tech stack

* **Vite 5** + **React 18** standalone SPA
* **TypeScript strict mode** — `strict: true`, `noUncheckedIndexedAccess`,
  `exactOptionalPropertyTypes` (see `tsconfig.json`)
* No SSR, no Next.js — operator tooling rarely needs SSR and the
  smaller surface keeps the threat model bounded.

## Postgres assumption

The console's read paths are implemented today by Postgres-backed
adapters:

* `flowforge-audit-pg` — provides `AuditSink.verify_chain()`.
* `flowforge-outbox-pg` — provides `OutboxRegistry.list_pending()`.
* The bundle's own SQLAlchemy models — provide instance lookups.

Non-Postgres hosts must wire equivalent adapters before deploying
this console. See `docs/v0.3.0-engineering-plan.md` §11 for the
PG-only assumption note.

## API contract

The console talks to the bundle's FastAPI backend via the typed client
in `src/api.ts`. The expected endpoints are:

| Method | Path | Backed by |
| --- | --- | --- |
| `GET` | `/admin/instances?jtbd=<id>&state=<s>&tenant=<t>` | bundle SQLAlchemy models |
| `GET` | `/admin/audit?topic=<t>&verify=true` | `AuditSink.verify_chain()` |
| `GET` | `/admin/saga/pending` | `flowforge.saga.queries.pending_compensations` |
| `POST` | `/admin/saga/<id>/compensate` | `flowforge.engine.fire(compensate=...)` |
| `GET` | `/admin/grants/history` | `AccessGrantPort.history()` |
| `GET` | `/admin/outbox?status=pending` | `OutboxRegistry.list_pending()` |
| `POST` | `/admin/outbox/<id>/retry` | `OutboxRegistry.requeue()` |
| `GET` | `/admin/rls/elevations` | audit chain topic `rls.elevate` |

The host wires these routes in its admin sub-app; the generated
backend exposes only the customer-facing routes. A reference
implementation lives in `flowforge-fastapi.admin`.

## Deployment notes

* Ship behind a separate ingress / hostname from the customer app.
* Require operator MFA at the auth proxy, not in the SPA.
* Set `VITE_ADMIN_API_BASE_URL` at build time to point at the admin
  back-end (default: `/admin`).
