## §1 Verdict

v2.1 is a genuine, non-regressive absorption of Appendix B point 3. The per-domain team model it defines is structurally sound: distinct version-controlled prompt artefacts, distinct citation-library extracts, and distinct reviewer agents per team — all under CI enforcement via `test_signoff_team_id_matches_registry` and `test_signoff_tier_consistent_with_package_suffix`. The Tier-A/B promotion path remains mechanically clean. The engineering critical path holds at 6 weeks, with E-77p correctly isolated in X2 under the schema-discipline reviewer and not touching the engineering worker's critical chain. Three bounded risks are introduced: (1) the registry.yaml single point of failure is real but mitigated through CI and quarterly audit; (2) the R-1 downgrade from High to Medium is mostly honest but elides within-team systematic blind-spot failure modes; (3) the 30-domain × 4-artefact maintenance surface is large relative to the quarterly drift-audit cadence. None of these rises to the BLOCKER level given the existing risk-register structure and the rollback discipline already in place from v2. The plan APPROVEs with one advisory item that the critic should note for §11.2 follow-up.

**Verdict: APPROVE**

## §2 Per-question evaluation

### Q1 — Does v2.1 materially absorb Appendix B point 3?

**Verdict: Yes — genuine absorption, not cosmetic relabelling.**

My Appendix B point 3 (`v0.2.0-plan-architect-review.md:348`) stated: "per-agent memory partitioning weakens [hallucination-correlation] if each persona is a distinct agent with a distinct citation-library prompt and distinct memory." v2.1 satisfies all three conditions concretely:

- Distinct system prompts: per-domain `author.md` and `reviewer.md` artefacts under `framework/python/flowforge-jtbd/team-prompts/<domain>/` (`v0.2.0-plan.md:151-152`), version-controlled, not runtime parameters. Drift is visible in `git blame` (`v0.2.0-plan.md:162`).
- Distinct citation-library extracts: each team consumes a per-domain slice of R-01 via `citations.yaml`; the reviewer agent is structurally prevented from seeing other domains' citations (`v0.2.0-plan.md:163`).
- Distinct reviewer agents: each domain has its own reviewer agent with domain-specific evaluation criteria; cross-domain hallucination is caught by specialised prompt, not a global reviewer (`v0.2.0-plan.md:164`).
- CI enforcement: `test_E_77p_citation_extract_validates_against_R01`, `test_E_77p_reviewer_rejects_vacuous_citation_in_pre_flight` (`v0.2.0-plan.md:225`) make the artefacts machine-verified, not aspirational.

The reviewer's "by design cannot fabricate cross-domain rules" claim (`v0.2.0-plan.md:152`) is architecturally sound for cross-domain pollution. My §6.4 demand for a per-domain citation library as a pre-X2 deliverable (`v0.2.0-plan-architect-review.md:200`) is satisfied: E-77p ships in X2 before X4, blocking v0.2.1 (`v0.2.0-plan.md:225`).

### Q2 — Is the Tier-A/Tier-B BLOCKER resolution preserved?

**Verdict: Yes — mechanically sound; no consumer confusion path.**

The V-1 BLOCKER from v0 (self-review masquerading as SME validation) was resolved in v2. v2.1 does not regress it. The machine-readable disclosure chain is complete and consistent across three layers:

- Package name suffix: Tier-B packages carry `-starter-extended` suffix; Tier-A use base name (`v0.2.0-plan.md:174`).
- Signoff schema: `tier: B`, `reviewer_credentials: ai-author`, `human_sme_review: pending` in YAML (`v0.2.0-plan.md:588-593`); `tier: A`, named human signer, `human_sme_review: complete` for Tier-A (`v0.2.0-plan.md:596-606`).
- New `team_id` field: `<domain>-team-B` vs `<domain>-team-A`, CI-validated against `registry.yaml` (`v0.2.0-plan.md:608`).
- README disclaimer: mandatory for all `-starter-extended` packages, templated at §15 (`v0.2.0-plan.md:559-580`).
- CI test `test_signoff_tier_consistent_with_package_suffix` (`v0.2.0-plan.md:608`) closes the consumer-confusion vector: a Tier-B package with `tier: A` in its signoff block would fail CI.

The promotion path is mechanical and additive (`v0.2.0-plan.md:392`): same team scaffold, same 4-agent structure, `team_id` updates from `-team-B` to `-team-A`, human SME role added, JTBDs re-signed without re-authoring. No re-architecture required. G-8 from the v2 critic APPROVE state is not weakened.

This is the key load-isolation claim. ### Q4 — Is the coordination model realistic for 30 teams?

**Verdict: Mostly yes, with a bounded weakness at the weekly council bandwidth.**

The four coordination mechanisms (`v0.2.0-plan.md:184-190`) are:

1. Weekly cross-domain council — package publishers from all active teams. At peak (v0.2.4–v0.2.7 with 5 concurrent teams per batch, potentially overlapping), this council could have up to 10 active teams represented simultaneously. With ≤6 concurrent flipping teams (1 release manager + ≤5 active, per `v0.2.0-plan.md:194`), the council has a bounded attendance ceiling. This is workable.
2. R-01..R-07 PR review by ≥2 affected teams — appropriate for shared-registry changes. Scales well because most R-01 changes affect 2–3 domains, not all 30.
3. E-47 lint-conflict solver in CI — machine enforcement at PR time; does not require human coordination bandwidth.
4. Single release manager — owns all teams' flip queue and F-7 cap.

The weakness is that the weekly council is the only human coordination mechanism between teams that are otherwise fully isolated. A cross-domain regulatory question (e.g., "does GDPR Art. 32 map to the same `compliance:` citation in healthcare and saasops?") waits up to 7 days for the next council meeting. At 30 domains this is probably acceptable given the isolation-by-design model. However, the plan does not specify how the council handles a quorum failure (e.g., a package publisher agent is misconfigured for a batch). This is a gap worth noting as an advisory, not a blocker, since the release manager role provides a backstop.

The single release manager as cross-team human bottleneck is a stronger concern at v0.2.4–v0.2.7 scale (20 Tier-B teams) than at v0.2.1–v0.2.3 scale (5 Tier-A teams). The plan does not address release manager capacity across 4 concurrent Tier-B batches. Advisory: §11.2 should add a release manager capacity note for v0.2.4+.

### Q5 — Is the per-team artefact maintenance surface bounded?

**Verdict: Adequately bounded, with a real but acknowledged single-point-of-failure.**

30 domains × 4 artefacts = 120 files plus `registry.yaml` as the 121st artefact (`v0.2.0-plan.md:445`). v2.1 acknowledges this as a new maintenance surface (`v0.2.0-plan.md:445-446`) without hand-waving it away.

Quarterly drift audits (§11.2.8, `v0.2.0-plan.md:398`) are the primary maintenance cadence. For AI-authored prompts this is plausibly sufficient: prompt files do not change unless someone edits them, and version control makes changes visible. The risk of silent drift is lower than with runtime prompt parameters, because a `git blame` audit is possible (`v0.2.0-plan.md:162`).

The `registry.yaml` single-point-of-failure is real: a missing team-id registration blocks a team's JTBD from passing CI (`v0.2.0-plan.md:399`). This is acknowledged as a "synchronisation point" in §12.5 (`v0.2.0-plan.md:446`). The mitigation — CI test `test_signoff_team_id_matches_registry` — makes the failure immediate and visible rather than silent. The release manager owns registry.yaml stewardship (`v0.2.0-plan.md:206`). These mitigations are appropriate for the risk level.

The one gap: the plan does not specify what happens to the registry.yaml when a Tier-B team is disbanded or its domain is deferred. Stale team-id registrations are not a correctness problem (existing JTBDs continue to validate), but they accumulate technical debt. This is advisory-level.

### Q6 — Is the R-1 risk downgrade from High to Medium justified?

**Verdict: Mostly honest; one failure mode is underweighted.**

The downgrade rationale in §9 R-1 (`v0.2.0-plan.md:347`) is: per-team isolation breaks the cross-domain hallucination-correlation surface. This is architecturally correct for cross-domain pollution. My Appendix B point 3 explicitly acknowledged this rebuttal. The downgrade is not wishful; it is technically grounded.

However, the downgrade does not address a failure mode distinct from cross-domain correlation: **within-team systematic blind spots**. A healthcare team's reviewer agent, operating exclusively from the `team-prompts/healthcare/reviewer.md` prompt and `citations.yaml` extract, has a systematic blind spot precisely because it is domain-isolated. If that reviewer prompt has a structural deficiency — e.g., it does not test for absence of HIPAA Safe Harbor de-identification documentation in JTBDs that involve PHI fields — every healthcare JTBD shipped by that team will carry the same deficiency without any cross-team signal catching it.

This failure mode is different from cross-domain correlation (which the team model does address) and is not meaningfully addressed by the within-team escalation loop, since the loop escalates from reviewer back to author using the same domain-isolated reviewer prompt.

The correct risk level for R-1 given the team model is therefore: cross-domain hallucination-correlation is legitimately downgraded; within-domain systematic reviewer blind spot is unchanged from the original concern. A composite R-1 of Medium is defensible if read as covering the aggregate, but it slightly overclaims the improvement. The plan should acknowledge this nuance explicitly. This is a should-fix advisory, not a BLOCKER.

## §4 New risks introduced by v2.1

**NR-1 — Within-team reviewer systematic blind spot (advisory)**
The team-isolation model that fixes cross-domain hallucination-correlation creates a new failure mode: a structural deficiency in a per-domain `reviewer.md` prompt propagates silently to every JTBD that team authors, with no cross-team signal catching it. The pre-flight test (`test_E_77p_reviewer_rejects_vacuous_citation_in_pre_flight`) validates that the reviewer rejects vacuous citations but does not validate that the reviewer's evaluation criteria are themselves complete. Mitigation: the quarterly prompt-drift audit (§11.2.8) should explicitly include a reviewer-criteria completeness check (not just a git-diff check for unexpected changes). This is not a new BLOCKER — R-1 at Medium remains defensible — but the nuance should be recorded in the risk register's R-1 row.

**NR-2 — Release manager capacity at Tier-B scale (advisory)**
At v0.2.4–v0.2.7 (4 Tier-B batches, 5 teams per batch), the single release manager chairs a weekly council with up to 10 concurrently active teams plus their F-7 flip queue. The plan does not specify a capacity bound or a handoff protocol if the release manager is unavailable for a council cycle. This is lower severity than a BLOCKER because the F-7 cap already limits per-minor churn, but the gap is worth tracking. Mitigation: §11.2 should add "release manager deputy designation for v0.2.4+ Tier-B batches."

**NR-3 — `registry.yaml` accumulation of stale team-ids (advisory)**
Team-ids registered in `team-prompts/registry.yaml` are never explicitly retired. Deferred or disbanded domain teams leave stale entries. Stale entries are not a correctness risk (CI validates that a JTBD's team_id exists in the registry, not the converse), but they create confusion at v0.3.0 planning when the registry is audited. Mitigation: add a `status: active | deferred | retired` field to registry.yaml entries, and validate at the quarterly drift audit.

## Consensus Addendum

**Antithesis (steelman against APPROVing):** The team isolation model is only as strong as the weakest per-domain reviewer prompt. If 25 Tier-B teams are activated with a common `_template/reviewer.md` scaffold and only minimal per-domain customisation (the plan does not specify the depth of per-domain differentiation required beyond "per-domain reviewer prompt"), then the team model is closer to "25 agents sharing a common reviewer template with domain-name substitution" than to "25 genuinely distinct regulatory review agents." In that case, the R-1 downgrade is overclaimed and the cross-domain isolation is more nominal than real. The plan should specify a minimum differentiation threshold for a reviewer.md to count as domain-specific (e.g., must reference ≥N domain-specific statutes in its evaluation criteria section).

**Tradeoff tension:** The team artefacts are version-controlled and audited quarterly, which is good for drift detection. But version-controlling 120 prompt files means that improving a reviewer prompt requires a PR, review by the release manager, and a council endorsement. This governance overhead may disincentivise prompt refinement when a team encounters a domain-specific edge case during authoring — the path of least resistance is to let a suboptimal reviewer prompt slide until the quarterly audit. The tension is between prompt quality improvement velocity and the governance discipline that prevents uncontrolled prompt drift. The plan resolves this tension conservatively (governance first), which is the right call for regulated content, but it should be explicit.

**Synthesis:** Accept the APPROVE. Add a one-line minimum differentiation criterion to §4.3.6 (e.g., "each `reviewer.md` must name ≥3 domain-specific statutory references not present in the `_template/reviewer.md`"), and add NR-1's within-team blind-spot note to R-1's mitigation column. Both changes are small enough to be incorporated as critic should-fixes rather than must-fixes.

**Principle violations (deliberate mode):** None at BLOCKER severity. P-2 (AI-author then domain-SME validate) is upheld: Tier-B reviewer is explicitly an AI agent, not a human, and the signoff schema makes this machine-readable. P-5 (every deliverable lands with conformance test green) is upheld: E-77p has 4 test IDs including a pre-flight validation. No principle is violated by the v2.1 additions.
