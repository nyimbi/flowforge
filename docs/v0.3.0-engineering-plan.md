# v0.3.0-engineering plan — generation pipeline improvements

Implementation plan for landing the 22 generation-pipeline improvements documented at [`docs/improvements.md`](improvements.md). Reached consensus through a 3-iteration RALPLAN-DR loop (Planner → Architect → Critic).

**Status**: APPROVED with 5 executor-facing residual notes. Ready to open W0 once the 4 pre-wave ADRs land.

**Track relationship**: This is the *engineering track*. It is **orthogonal** to the existing content-track [`v0.2.0-plan.md`](v0.2.0-plan.md), which targets E-74..E-82 per-domain JTBD content delivery. The two tracks proceed in parallel; this plan ships under whichever minor the maintainers select (defaulting to `v0.3.0`).

---

## 1. Principles

These five principles are non-negotiable. Every wave, every PR, every reviewer enforces them.

1. **Determinism is non-negotiable.** Every new generator must produce byte-identical output across runs. Items that cannot preserve this (item 22 LLM polish) run *outside* the regen pipeline as a separate authoring step.
2. **Per-bundle generators must be aggregations.** Items spanning JTBDs (8 OpenAPI, 11 lineage, 16 analytics, 17 i18n, 18 design tokens) emit one cross-cutting artifact per bundle, not per-JTBD slices. Per-JTBD code stays per-JTBD; per-bundle code stays per-bundle. No hidden coupling.
3. **Audit-2026 invariants don't regress.** Conformance suite + 4 ratchets + signoff gate stay required-green for every PR. No new generator may add an I/O dep to `flowforge-core`. Cross-runtime parity stays green for any expr-touching change.
4. **Hexagonal port discipline for new runtime capabilities.** Items 12 (OTel), 16 (`AnalyticsPort`), 6 (idempotency dedupe storage) become *ports*, not direct imports. Adapter packages live alongside existing ones. In-memory fakes ship in `flowforge.testing.port_fakes`.
5. **Determinism before correctness, in tests.** The first signal an item works is `examples/*/generated/` regenerating identically; behavioural tests come second. This pins regen-diff before chasing functional correctness.

---

## 2. Decision drivers

1. **Sequencing minimises rework on shared file surfaces.** Items 13 (form), 16 (analytics hooks), 18 (tokens), 21 (visual-regression baselines) all share `Step.tsx.j2`; they sequence 13 → 16, 13 → 18 → 21. Items 1 (migration safety), 6 (idempotency table), 7 (restore runbook) all share `db_migration` generator and benefit from clustering.
2. **Wave shape balances reliability and visibility per release.** A "reliability-first / UX-last" split leaves demos stale for months. Each wave mixes reliability and visibility so progress shows on both axes.
3. **Port additions cluster within a single wave.** New runtime ports must land before generators that import them. W2's internal sub-batching (observability backbone before consumers) prevents half-adopted hosts from running with stub spans.

---

## 3. Selected option

**Option B** — dependency-respecting waves with feature flags where API-shape risk demands it, sidecar artifacts for non-deterministic authoring (item 22), and a budgeted cumulative-invariant gate.

### Alternatives considered

**Option A — sequential by section** (Reliable 1-7 → Capable 8-12 → Functional 13-17 → Beautiful 18-22). Rejected: breaks the front-end dependency chain (13 → 18 → 21, 13 → 16) across sections, forcing rework when downstream items land.

**Option C — slice by infrastructure layer** (ports → generators → templates → CI gates → CLIs). Rejected: front-loads invisible infrastructure with no visible delta for months. Pairs items that are most useful together (item 13 real form + item 21 visual regression) into different phases.

**No-wave per-item flag-gating** (Architect's antithesis). Retained as **fallback**: if two consecutive waves block on a single slow item, drop the wave abstraction and land items independently behind `bundle.flags.<item>` toggles, default off.

## 5. Pre-mortem

Three scenarios. Each mitigation is CI-enforceable.

### Scenario 1 — Silent determinism drift in unexercised generators

A new generator (e.g. item 18 design tokens) introduces dict iteration that's incidentally ordered today and breaks on a different Python build. `scripts/check_all.sh` step 8 catches it on regen, but only against bundles that exercise the new field. If `bundle.project.design` is set on no example, the regression sits dormant until production adoption.

**Mitigation (CI-enforceable)**: bidirectional fixture-registry test at `tests/audit_2026/test_generator_fixture_coverage.py`.

- *Forward*: each generator declares `CONSUMES = ("project.design", ...)` in `flowforge_cli/jtbd/generators/_fixture_registry.py`. Test asserts ≥1 example bundle populates each declared field.
- *Reverse*: AST-walk every generator under `flowforge_cli/jtbd/generators/*.py`, collect every `bundle.<field>` and `jtbd.<field>` attribute access, assert each is declared in `CONSUMES`. Failure message names the generator and the undeclared path.
- *Escape hatch*: `# fixture-registry: <field>` comment for dynamic patterns (`getattr`, `model_dump`); adding one requires an ADR.

### Scenario 2 — Cross-runtime parity break (invariant 5)

Item 13's `show_if: context.<edge_id>` is an expression evaluated in both Python (server-side gate) and TypeScript (client-side conditional render). Without the 200-input fixture being extended, Python and TS evaluators silently drift on `show_if`-shaped expressions; conformance invariant 5 fails on production.

**Mitigation (CI-enforceable)**:
- New fixture `tests/cross_runtime/fixtures/expr_parity_v2.json` (250 cases: 200 existing + 50 `conditional`-tagged) lands in same PR as item 13's first `show_if` emission.
- Existing `expr_parity_200.json` retired in W3 after `expr_parity_v2.json` proves stable for two waves.
- New ratchet `scripts/ci/ratchets/no_unparried_expr_in_step_template.sh`: greps `Step.tsx.j2` for any expression-shaped token; fails CI if the fixture wasn't updated in the same commit.
- PR template gets explicit "Touches expr evaluator?" checkbox.

### Scenario 3 — Alembic chain conflict on already-deployed hosts

Items 1, 6, 7 emit fresh migrations. The deterministic revision id (`sha256(package + jtbd_id)[:12]`) is stable, but if a host has already deployed an older bundle version, the new migration's `down_revision` doesn't align with the host's `alembic_version` table. Deploy fails with `multiple heads`.

**Mitigation (CI-enforceable)**:
- Item 1's safety analyzer detects multi-head conditions in deployed chains.
- `flowforge pre-upgrade-check --alembic-chain` runs against the host's deployed `alembic_version` before upgrade; refuses to proceed on multi-head.
- Cumulative-invariant gate's downgrade-then-upgrade rehearsal subtarget: for each wave's head migration, `alembic downgrade <prev>` then `alembic upgrade head` against testcontainers Postgres, asserting both directions complete and audit chains re-verify.

## 6. Expanded test plan

Eight layers + cumulative gate. Coverage required for every wave.

| Layer | Coverage |
|---|---|
| **Unit** | Each new generator has a generator-test (input bundle → expected file content, byte-identical). New transforms (idempotency dedupe, compensation synthesis) get isolated unit tests. |
| **Integration** | `tests/integration/python/` runs the generator against each example bundle, then exercises the resulting app: boot FastAPI, fire canonical event, verify audit chain, verify outbox dispatch. New ports get integration tests with the real adapter (testcontainers Postgres for OTel + idempotency; in-memory for analytics). |
| **E2E** | `tests/integration/e2e/` adds a flow per example: form submit → audit emit → outbox dispatch → notification delivery. With item 13 (real form), this becomes an actual Playwright form-submission test. With item 6 (idempotency), the e2e fires the same event with the same key twice and asserts only one fire. |
| **Observability** | New OTel spans assert via OTel test harness that spans carry expected attributes. `flowforge audit-2026 health` CLI gets new SLI probes per wave. PromQL alert rules in `tests/observability/promql/` add rules for the new ports. |
| **Cross-runtime parity** | Fixture v2 (250 cases). `make audit-2026-cross-runtime`. PR template surfaces the obligation as a checkbox for any expr-touching item. |
| **Determinism** | Every wave runs `scripts/check_all.sh` step 8 byte-identical regen against all example bundles + every flag-value combination (e.g. `form_renderer = "skeleton" | "real"`). Hash input is the `(bundle, sidecar)` tuple, not just the bundle. |
| **Conformance** | Every wave runs `make audit-2026-conformance` with all current invariants required-green. New invariants land per wave: 10 (compensation symmetry, W0), 11 (idempotency-key uniqueness, W2b). |
| **Property** | W4a-onward: every generator added in W0-W3 has a hypothesis property test with a pinned seed (per ADR-003). |
| **Cumulative gate** | `make audit-2026-cumulative` runs every prior wave's invariants on the current wave's PR. **Time budget**: 5 minutes per PR. **Blocking subset**: invariants 1, 5, 10, 11 (P0) block PR merge regardless of budget; P1 invariants overflow to nightly via `make audit-2026-cumulative-smoke` (P0 only). |

## 9. Per-wave rollback

Each wave's rollback runbook lives at `docs/v0.3.0-engineering/rollback-W<n>.md` with structured frontmatter:

```yaml
---
wave: W2
incompatible_with_rollback_of: [W1]   # rolling back W1 with W2 still applied breaks W2's FK to W1.idempotency_keys
migrations:
  - rev: <hash>
    table: idempotency_keys
    fk_dependencies: [<jtbd>_table.tenant_id]
flag_flips:
  - bundle.project.frontend.form_renderer: "real" -> "skeleton"
revert_shas:  # filled in at wave merge
  - <SHA>: "item 6 idempotency keys"
---
```

The cumulative-invariant gate's downgrade-then-upgrade rehearsal validates each wave's head migration is reversible. Rollback runbooks are templated; the reversal is a `git revert <SHA>` series followed by `alembic downgrade` per migration.

## 10. CI shard plan

Per-PR CI grows ~5 min worst case from the cumulative gate. To stay under budget:

- `make audit-2026-property` and `make audit-2026-cross-runtime` shard 4-way via `pytest-xdist`.
- Visual regression (item 21) runs **nightly**, not per-PR. Per-PR runs the smoke subset (DOM-snapshot only on the canonical example).
- SLA stress harness (item 5) runs nightly; not per-PR.
- LLM copy polish (item 22) is opt-in; never runs in CI without an explicit `--ci` flag.
- Visual regression baseline rebases happen weekly via a scheduled workflow that opens a PR with the new baselines.

## 11. Executor residual risks

These are implementation traps the executor needs to know about. None require plan revision; all are concrete fixes the executor applies during implementation.

1. **`test_expr_parity.py` requires line-level edits, not pure addition.** When item 13 ships fixture v2: edit `:23` (FIXTURE_PATH constant), `:33-35` (count assertion), `:50-63` (required-tags set adds `"conditional"`). Plan calls out fixture v2 but the executor must not miss these specific lines.
2. **Bidirectional AST registry must handle dynamic access.** `getattr(bundle, name)`, `**bundle.model_dump()`, and computed attribute paths produce false negatives in the reverse-coverage walk. Convention: an explicit `# fixture-registry: <field>` escape-hatch comment is allowed; adding one requires an ADR. Without this convention, the registry will block legitimate dynamic patterns and erode trust.
3. **Sidecar must enter regen-diff hash input.** Two bundles with identical `spec_hash` may emit divergent forms if their sidecars differ. `scripts/check_all.sh` step 8 must hash the `(bundle, sidecar)` tuple, not just the bundle. Otherwise a sidecar drift passes the hash check and fails visual-diff much later.
4. **Cumulative-gate "blocking subset" needs explicit listing.** Make it explicit in the Make target description: invariants 1, 5, 10, 11 (the P0 set) block PR merge regardless of budget; overflow shifts only the P1 invariants to nightly.
5. **Sidecar lookup precedence needs a spec.** Search order: (1) explicit `--overrides <path>` flag; (2) co-located `<bundle_path>.overrides.json`; (3) none, use canonical strings. Two examples nested in the same parent dir cannot cross-pollinate.

## 12. ADR

**Decision**: Land 22 generation-pipeline improvements via 6 dependency-respecting waves (W0 + W1 + W2 + W3 + W4a + W4b), behind feature flags where API-shape risk demands it (item 13), with override authoring as a sidecar artifact (item 22), 4 pre-wave principle pins, 2 new conformance invariants, a budgeted cumulative-invariant gate (5 min/PR; overflow to nightly), and per-wave rollback runbooks declaring `incompatible_with_rollback_of:`.

**Drivers**:
1. Sequencing minimises rework on shared file surfaces (`db_migration.py`, `Step.tsx.j2`, `transforms.derive_transitions`).
2. Wave shape balances reliability and UX visibility per release.
3. Observability backbone (W2a) lands before its consumers (W2b) within W2.

**Alternatives considered**:
- *Option A* (sequential by section) — rejected; breaks the front-end dependency chain (13 → 18 → 21, 13 → 16) across sections.
- *Option C* (slice by infrastructure layer) — rejected; front-loads invisible infrastructure with no visible delta for months.
- *No-wave per-item flag-gating* — retained as fallback if waves block.

**Why chosen**: Best balance of dependency-respect, visibility per release, and bounded scope per wave. Concrete CI gates and rollback runbooks per wave preserve audit-2026-style hardening discipline.

**Consequences**:
- Hosts adopting v0.3.0 must opt into `form_renderer = "real"` to access W3 visuals; existing examples remain on `"skeleton"` until W3 visual baselines stabilise. Removal of `"skeleton"` is scheduled for v0.4.0 / v1.0.0.
- New runtime ports land in W2 (`MetricsPort` extension, `AnalyticsPort`, `TracingPort`) — host wiring required.
- `z3-solver` is a runtime extra; reachability artifacts are skipped without it.
- Override authoring is sidecar-based; canonical bundle remains untouched; `spec_hash` invariant preserved.
- Per-PR CI grows ~5 min worst case from cumulative-invariant gate; overflow runs nightly.

**Follow-ups**:
- Stale docstring fix at `tests/conformance/test_arch_invariants.py:9` — lands in W0.
- Decision on minor-version cadence (one tag for all 6 waves vs. per-wave minor) — defer to release-manager call after W0.
- `expr_parity_200.json` retired in W3 after `expr_parity_v2.json` proves stable.
- DRI assignments at sprint start (placeholder `Owner: TBD` per item).
