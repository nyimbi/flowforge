## Context

Item 21 proposes a visual-regression CI gate: render every generated frontend page through Playwright at three viewports and commit baseline screenshots to `examples/<example>/screenshots/`. The naive implementation compares **pixel bytes** of screenshots against baselines.

Pixel bytes are non-deterministic across Chromium minor versions even with pinned binaries. Font hinting, GPU compositing, antialiasing, and subpixel rendering all drift between Chromium releases, between Linux kernel versions, and between CI runners with different glibc versions. A pixel-byte equality gate would generate false-positive failures on every Chromium version bump in the runner image.

Principle 1 (determinism non-negotiable) is at stake: any CI gate that produces non-deterministic results across hosts breaks the regen-diff invariant the framework rests on.

## Decision

**Two artifacts, two gates, two cadences.**

1. **DOM snapshots** are the **CI-gating** artifact. Per page, Playwright extracts the rendered DOM, normalises it (whitespace, attribute order, comment stripping), and writes `<example>/screenshots/<page>.dom.html`. The byte-identity test runs on every PR and is required-green for merge. This is what `make audit-2026-visual-regression-dom` runs.

2. **Pixel screenshots** are **advisory**. Per page, Playwright captures a PNG and writes `<example>/screenshots/<page>.<viewport>.png`. CI runs an SSIM (Structural Similarity Index) comparison against the baseline with threshold ≥ 0.98. This is what `make audit-2026-visual-regression-ssim` runs. The advisory gate runs **nightly**, not per-PR, and posts results as a PR comment when relevant; it does **not** block merge.

3. **Per-PR smoke**: a subset of DOM snapshots (the canonical `examples/insurance_claim/` page only) runs per-PR; the full DOM-snapshot suite runs nightly.

## Why DOM snapshots are deterministic

The DOM tree is generated by React from the `form_spec.json` and `Step.tsx` template. Both are deterministic outputs of the generator. Playwright's DOM serialisation (after normalisation) produces byte-identical bytes for byte-identical inputs across Chromium versions. The only non-determinism in the rendering pipeline lives in the GPU compositor and font rasteriser, both of which produce pixels but not DOM nodes.

## Why SSIM ≥ 0.98 is the right threshold

SSIM is the industry-standard perceptual similarity metric. 0.98 corresponds to "perceptually identical" — humans cannot tell two SSIM-0.98 images apart at normal viewing distance. Lower thresholds (0.95 or below) admit visually different layouts; higher thresholds (0.99+) flake on font-hinting drift.

## Consequences

**Positive:**
- DOM-snapshot byte-identity preserves Principle 1 across Chromium versions.
- Pixel screenshots remain available for human review (committed to the repo) without blocking merges on rendering drift.
- CI cost stays bounded: per-PR runs only the DOM smoke; nightly runs the full suite.

**Negative:**
- DOM-snapshot equality misses purely visual regressions (e.g. a CSS `color` change that's syntactically identical in DOM). The advisory pixel gate catches these but does not block.
- The advisory gate's failure-handling discipline (who triages pixel diffs that exceed the threshold) is a process question, not a technical one. Default: advisory failures get filed as issues but don't block release.
- Pixel screenshots are committed; they bloat the repo. Mitigation: store under Git LFS once the screenshot count exceeds 100.

**Neutral:**
- Baseline rebases happen weekly via a scheduled GitHub Action that opens a PR with new baselines. Reviewer judgement decides whether the new baselines are intentional.

## Alternatives considered

1. **Pixel-byte equality only** — rejected, non-deterministic across Chromium versions. Would generate false positives on every runner-image update.
2. **SSIM ≥ 0.98 as the CI gate** — rejected, the SSIM library itself is non-deterministic at sub-threshold variance levels (different SSIM implementations produce different scores on identical inputs). Acceptable as advisory, not as a gate.
3. **Percy / Chromatic-style hosted visual diff** — rejected, adds a third-party dependency with its own outage modes. Reconsider when the team has bandwidth to evaluate.
4. **Skip visual regression entirely** — rejected, item 21's value (catching invisible-until-production regressions in generated frontends) is real. The trade-off is between "no gate" and "deterministic gate"; we chose the latter.

## Implementation notes

- DOM normalisation rules: strip `data-react-*` attributes, normalise whitespace, sort `class` token list alphabetically, sort element attributes alphabetically.
- Baseline directory layout: `examples/<example>/screenshots/{<page>.dom.html, <page>.<viewport>.png}`.
- Baseline rebase workflow: `.github/workflows/visual-baselines-rebase.yml` (weekly cron).
- Threshold tuning: 0.98 is the starting point; if false-positive rate exceeds 5% in the first month, drop to 0.95 with a documented rationale.
