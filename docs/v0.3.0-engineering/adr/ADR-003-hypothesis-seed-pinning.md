## Context

Item 3 generates a hypothesis property suite per JTBD. Hypothesis is RNG-driven: each test run draws inputs from a strategy, and on failure performs shrinking to find a minimal counter-example.

Hypothesis's defaults are not deterministic across CI hosts:

- `Settings.derandomize=False` (default) seeds from `os.urandom`, producing different inputs per run.
- Even with `derandomize=True`, the **shrink** behaviour can diverge across hypothesis versions and across CI shards (different shrink paths produce different "minimal" failures).

Principle 5 (determinism before correctness in tests) requires that the same generator → same property test → same input space → same outcome across CI hosts. Without seed pinning, two CI shards running the same property test can report different counter-examples for the same bug, eroding the trust the framework rests on.

## Decision

**Every emitted property test pins its seed deterministically from the JTBD id.**

`hypothesis.seed` is a separate decorator from `hypothesis.settings` in hypothesis 6.x — `settings(seed=...)` is not a supported kwarg on the published API. Stacking the two decorators reproduces the ADR's intent on the actual API surface:

```python
# Generated by flowforge_cli.jtbd.generators.property_tests
import hashlib
from hypothesis import given, seed, settings, strategies as st

JTBD_ID = "claim_intake"
_SEED = int(hashlib.sha256(JTBD_ID.encode()).hexdigest()[:8], 16)

@given(...)
@seed(_SEED)
@settings(derandomize=True, max_examples=200)
def test_claim_intake_workflow_terminates(...):
    ...
```

For readability, the generated source typically lifts the two decorators into module-level constants once per file:

```python
_PROP_SEED = seed(_SEED)
_PROP_SETTINGS = settings(derandomize=True, max_examples=200, deadline=None, ...)

@given(...)
@_PROP_SEED
@_PROP_SETTINGS
def test_claim_intake_workflow_terminates(...):
    ...
```

The behaviour is identical to applying the decorators directly.

### Why `sha256(jtbd_id)[:8]` as int

- **Deterministic**: same JTBD id produces the same 32-bit seed across hosts.
- **Per-JTBD distinct**: two JTBDs in the same bundle don't share input spaces, so a flaky test in one doesn't shadow a real bug in another.
- **Stable under refactor**: the seed is derived from the JTBD's identity, not from file paths or test names. Renaming a test does not move the seed.
- **32-bit**: hypothesis seeds are accepted as Python ints; 32 bits is plenty of entropy for input space exploration without overflowing the default seed type.

### `derandomize=True`

Combined with the explicit seed, this disables the "fresh random per run" behaviour. The same test produces the same draw sequence every time. Shrink behaviour also stabilises (within a hypothesis version).

### `max_examples=200`

Default is 100. We want stronger coverage for generator-emitted tests since they exercise synthesised state machines that humans don't review case-by-case. 200 is a starting point; tunable per generator-test if needed.

### Hypothesis version pinning

The `flowforge-cli` runtime extra (already declared in [`dependency-groups.dev`](../../../pyproject.toml)) gets a hard pin: `hypothesis>=6.100,<7.0`. Cross-version shrink-behaviour drift is the dominant remaining non-determinism source; pinning to a major version bounds it.

## Consequences

**Positive:**
- Two CI shards running the same property test produce the same counter-example on failure. Bug reports are reproducible.
- Adding a new JTBD to a bundle does not perturb the seeds of existing JTBDs.
- Failures are debuggable locally: `pytest -k <test_name>` produces the same failure as CI.

**Negative:**
- The seed is fixed once per JTBD id. If the bug is sensitive to a specific input not in the deterministic draw sequence, hypothesis won't find it. Mitigation: `max_examples=200` widens the search; rare bugs may need explicit `@example(...)` annotations.
- `sha256(jtbd_id)[:8]` is collision-prone in theory (32-bit space). In practice, JTBD ids are human-readable strings that rarely collide on 32 bits, but the property-coverage gate checks for collisions and refuses to emit duplicate seeds.

**Neutral:**
- Hosts running the test suite locally see the same failures as CI. This is the desired outcome.

## Alternatives considered

1. **`derandomize=False` (hypothesis default)** — rejected; non-deterministic across runs. Defeats the purpose of CI gating.
2. **Fixed integer seed (e.g. `seed=42`) for every test** — rejected; all JTBDs share an input space, so a flaky test in one shadows real bugs in others.
3. **Hash full JTBD spec_hash for the seed** — rejected; the spec changes when content changes, and the seed should be stable across content edits (we want to find the *same* bug with the *same* input even if the spec evolves).
4. **Per-test seed pinned to the test name** — rejected; renaming a test would move the seed, eroding the "rename is a no-op" property.
5. **Property tests run only nightly, not per-PR** — rejected; defeats the goal of catching bugs at PR time.

## Implementation notes

- Generator: `python/flowforge-cli/src/flowforge_cli/jtbd/generators/property_tests.py`.
- Template: `python/flowforge-cli/src/flowforge_cli/jtbd/templates/property_tests/test_<jtbd>_workflow.py.j2`.
- The `_SEED` constant is computed at template-render time, not at test-run time. This means the seed is visible in the generated source and easy to grep.
- Collision check: `tests/audit_2026/test_hypothesis_seed_uniqueness.py` walks every emitted seed and asserts no two JTBDs in the same bundle share one.
