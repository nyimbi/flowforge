## 1. RALPLAN-DR Summary

### 1.1 Five guiding principles

| # | Principle | Why it matters here |
|---|---|---|
| P-1 | **Reuse the audit-2026 quality template for every new JTBD.** | We already enforce data_capture ≥3 distinct fields, edge_cases ≥1, documents_required ≥1, approvals ≥1, notifications ≥1, sla, data_sensitivity, compliance, signoff metadata. This is the bar — any drift creates a two-tier catalogue and breaks the §4.2 D-01 lint rule. |
| P-2 | **AI-author then domain-SME validate.** | 875 net-new JTBDs cannot be hand-written by 30 SMEs in calendar weeks. The hybrid pattern: AI agent generates a structured first draft per domain (anchored to a per-domain template + regulatory citation list); a named domain SME reviews each yaml and signs the `signoff` block. Same pattern that produced the 5 strategic verticals in audit-2026 S2b, scaled. |
| P-3 | **Parallelize across independent domain workers; coordinate only on shared schema.** | 30 domains touch 30 disjoint package directories. The shared bottleneck is the schema (`flowforge_jtbd.dsl.spec.JtbdLintSpec`) — freeze it for the duration; everything else runs in parallel. |
| P-4 | **Engine wiring (E-74) ships behind a feature flag; never regress audit-2026 conformance invariants.** | Conformance invariants 2 (engine atomicity), 3 (replay determinism), 4 (saga durability) are SOX/HIPAA-grade after audit-2026. Touching `engine/fire.py` requires `forks_enabled = False` default-off for one minor + every PR runs the full conformance suite required-green. |
| P-5 | **Every deliverable lands as a PR with conformance test green.** | No "land then test" — each engineering ticket and each domain JTBD lands with its acceptance test alongside it. The signoff CI gate already enforces this for audit-2026 tickets; extend it to v0.2.0. |

### 1.2 Top three decision drivers

| # | Driver | Implication |
|---|---|---|
| D-1 | **JTBD content is the largest single scope item — ~875 net-new JTBDs across 30 domains.** | This dwarfs every engineering ticket combined. Plan must front-load the content pipeline and parallelise it heavily; engineering tracks must not block it. |
| D-2 | **Security follow-ups (E-73 phases 4–6 + real JWT extractor + per-fix metric emitters) require DELIBERATE-mode signoff.** | Per-user audit identity, JWT verification, and metric emission all touch security boundaries. They get signoff-checklist rows + observability validation just like audit-2026 P0/P1 fixes. |
| D-3 | **E-74 phases 2–5 wires `parallel_fork` through `fire()`; this is high-risk to invariants 2/3/4.** | `engine/fire.py` carries the entire audit-2026 hardening (E-32 EPIC + E-39 + E-40). Touching it without a feature flag risks regressing P0 fixes. Mitigation: strict feature-flag-default-off, per-PR invariant gate, single-EPIC-PR convention. |

### 1.3 Three viable options with pros/cons

| Option | Description | Pros | Cons | Decision |
|---|---|---|---|---|
| **A — Sequential by capability** | Engineering tracks (X1: metrics + JWT + E-73) → Content track (X2: 875 JTBDs) → Engine wiring (X3: E-74). One thing at a time. | Simpler coordination; content team picks up after schema is stable. | Total elapsed time ~12 weeks; blocks content track for ~2 weeks while engineering finishes X1; misses the parallelism the user explicitly named ("executed by teams in parallel"). | Rejected. |
| **B — Parallel tracks with shared schema lock (CHOSEN)** | X1 engineering + X2 content + X3 engine wiring run concurrently from week 1. Schema is frozen at sprint kick-off; engineering changes that would shift schema are blocked behind X3 close. | Matches user's "by teams in parallel" guidance; ~10 calendar weeks; uses the audit-2026 multi-team pattern that closed 77 findings in one session. | Requires schema-freeze discipline; coordination overhead between 30+ workers; need 30 SME-equivalent reviewers (can be the same Claude agent acting in domain-SME role with citation discipline). | **Chosen.** |
| **C — Content-only sprint, defer all engineering** | Ship the 875 JTBDs as v0.2.0; defer E-73/JWT/metrics/E-74 to v0.3.0. | Lowest engineering risk; one cohesive content release. | Leaves the audit-2026 follow-ups stranded; "publishable to PyPI" gets the content but not the security identity story; user explicitly asked for the engineering work. | Rejected — violates explicit user scope. |

**Choice**: Option B (parallel). Driven by P-3 (parallelize across independent workers), D-1 (content dominates), and the user's "executed by teams in parallel" instruction.

SME signoff is gated on citation accuracy, not just schema. | **F-2** | **E-74 engine wiring regresses conformance invariants 2/3/4** (engine atomicity, replay determinism, saga durability). | `engine/fire.py` is touched by E-32 EPIC + E-39 + E-40. Adding fork dispatch + per-token advance + join barrier mutates the same code paths. | `pytest framework/tests/conformance/test_arch_invariants.py` fails on PR; CI gate rejects merge. | Feature flag `forks_enabled = False` default-off for v0.2.0; flip default-on in v0.3.0 only after one minor of opt-in soak. Single-PR EPIC for E-74 (per F-1 of audit-2026); CODEOWNERS lock on `engine/fire.py`. Required-green on invariants 2, 3, 4 every PR. | `git revert <epic-sha>`; flag stays off; existing host-managed pattern (WorkflowInstanceToken + context flags) keeps working. | Engine-track owner |
| **F-3** | **875 new JTBDs balloon CI time — `make audit-2026-unit` slows from <30s to >5min**, breaking PR feedback loop. | Each new JTBD adds smoke-test cases; 30 × 30 JTBDs × N tests = explosion. | CI duration >5 min on PR; smoke-test wall-clock dashboards trip 5× threshold. | Per-domain CI sharding (matrix job per domain); domain pkgs stay `package=false` until reviewed (E-46 / F-5 two-step rollout already in place, just extends to all 30); skip-pattern in `make audit-2026-unit` for non-strategic domain pkgs unless their test files changed. | Disable cross-domain CI; revert the `package=true` flips one at a time. | CI/test eng |
| **F-4** | **Real JWT extractor (E-76) introduces a JWT-spec corner case the test suite misses** (e.g. unverified algorithm, kid mismatch, expired-but-near). | We're rolling our own JWT verification on top of `flowforge-signing-kms` instead of using a battle-tested lib. | Security-team review or a fuzz-test campaign post-merge surfaces the bug. | Don't roll our own JWT parser. | **F-5** | **Per-fix metric emitters drift between code and PromQL alert rules** — alerts reference metric names that no code emits, so alerts never fire. | The PromQL rules at `framework/tests/observability/promql/audit-2026.yml` reference 12 metric names; today only 3 of them are actually emitted. Adding emitters without a ratchet means future drift. | `flowforge audit-2026 health` returns "no data" for tickets that should have signal. | Add a CI test that parses both the PromQL alert file AND the source tree (grep for metric names), asserts every alerted-on metric is emitted somewhere. New ratchet `scripts/ci/ratchets/no_orphan_promql_metrics.sh`. | Comment out the PromQL rule pending real metric; track in backlog. | Observability/SRE eng |

## 11. ADR — Architecture Decision Record

### 11.1 Decision

Adopt **Option B (parallel tracks with shared schema lock)** to ship FlowForge v0.2.0 in 10 calendar weeks. Engineering tracks (X1, X3, X4) run sequentially on the engine critical path; the content track (X2) runs parallel from week 1 with 30 SME-domain workers and a schema-discipline reviewer.

### 11.2 Drivers

1. **Content dominates scope** (D-1): ~875 net-new JTBDs cannot be hand-authored serially.
2. **Security follow-ups need DELIBERATE signoff** (D-2): E-73 phases 4–6, E-76 JWT, E-75 metric emitters.
3. **Engine wiring is high-risk** (D-3): E-74 must not regress invariants 2/3/4.

### 11.3 Alternatives considered

| Option | Reason for rejection |
|---|---|
| A — Sequential by capability | ~12 wk total; blocks content for 2 wk; misses user's "parallel teams" guidance. |
| C — Content-only sprint | Leaves engineering follow-ups stranded; user explicitly asked for them. |

### 11.4 Why chosen

- Aligns with user's "executed by teams in parallel" directive.
- Reuses the audit-2026 multi-team pattern that closed 77 findings in one session.
- Engine work is bounded (1 worker × 3 wks) and does not block content.
- 30 domain workers run independently — no cross-coordination beyond the schema lock.

### 11.5 Consequences

**Positive**:
- v0.2.0 ships in ~10 wk with both content depth and engineering follow-ups closed.
- Content track produces 900+ JTBDs, fulfilling the user's exhaustive-coverage requirement.
- Engine wiring lands behind feature flag — zero regression risk to v0.1.x deployments.

**Negative**:
- Coordination overhead across 30 SME-domain workers; need a strong schema-discipline reviewer.
- AI-authored content has factual-error risk requiring SME review pass.
- E-74 EPIC concentrates merge risk in week 7–9; requires CODEOWNERS lock + single-PR discipline.

**Mitigations**: All five risks (R-1..R-5) have named owners + concrete actions + rollback paths.

### 11.6 Follow-ups

- Post-v0.2.0: flip `forks_enabled = True` default in v0.3.0 after one minor of opt-in soak.
- Post-v0.2.0: open backlog ticket for the cross-domain JTBD-discovery API (currently each domain pkg is independent).
- Quarterly: re-run AI-authored content through SME review to catch regulatory drift (laws change).

## 13. Open questions for architect

1. Is the AI-author + SME-validate pattern strong enough for regulated-domain content (e.g. healthcare HIPAA timelines, banking BSA SAR deadlines), or do we need a third pass (legal review)?
2. Should `forks_enabled` be a per-instance flag (workflow opts in) or a global engine flag? Per-instance is safer but adds complexity.
3. Is the 30-JTBDs-per-domain target floor or ceiling? User said "minimum of 30 — preferably exhaustive". For some domains exhaustive could be 60+ (e.g., insurance lines of business); for others 30 may be aspirational. Per-domain target locked at sprint kickoff?
4. Which tier-1 vertical gets the first content top-up sprint (E-77)? Suggest insurance because it already has the deepest audit-2026 baseline (6 JTBDs incl. reins_place).
5. Should v0.2.0 release include a "content-only" PyPI sub-release (just the 30 jtbd-* packages flipped to `package=true`) ahead of the engineering work, or all-or-nothing?
