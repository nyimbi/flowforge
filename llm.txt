# Flowforge LLM Guide

This file is the root entrypoint for AI agents working with Flowforge. Prefer
this file first, then [README.md](README.md), then
[docs/jtbd-user-guide.md](docs/jtbd-user-guide.md), then package-specific
READMEs.

## What Flowforge Is

Flowforge is a portable workflow framework for regulated, multi-tenant business
applications. It has:

- a JSON workflow DSL;
- a pure Python workflow engine;
- host-wired ports for infrastructure;
- adapter packages for FastAPI, SQLAlchemy/Postgres, tenancy, audit, outbox,
  RBAC, documents, signing, notifications, and observability;
- TypeScript packages for rendering, runtime clients, a visual workflow
  designer, and JTBD editing;
- a deterministic JTBD-to-application generator.

UMS is not a Flowforge dependency. Use UMS only when an explicit downstream
release-certification task asks for UMS parity.

## Setup

From the repository root:

```bash
make setup
```

Equivalent:

```bash
bash scripts/setup.sh
```

CLI-only fallback:

```bash
uv sync
uv run flowforge --help
```

Do not invent install commands. This repo is source-first and not yet a single
published PyPI install target.

## Main Commands

```bash
uv run flowforge tutorial --out /tmp/flowforge-demo --no-pause
uv run flowforge jtbd lint --bundle examples/insurance_claim/jtbd-bundle.json --warn-only
uv run flowforge jtbd-generate --jtbd examples/insurance_claim/jtbd-bundle.json --out /tmp/flowforge-insurance --force
uv run flowforge validate --def /tmp/flowforge-insurance/workflows/claim_intake/definition.json
uv run flowforge simulate --def /tmp/flowforge-insurance/workflows/claim_intake/definition.json --events submit
```

Quality gates:

```bash
bash scripts/check_all.sh
make audit-2026-release-local
make audit-2026-release-external-preflight
```

The external release gate is manual release evidence. It needs browser support,
reviewed sidecar evidence, downstream UMS parity only when explicitly
configured, and live Postgres.

## JTBD Bundle Grammar

Bundles are JSON or YAML.

Top-level:

```yaml
project: {}
shared: {}
jtbds: []
```

`project` requires:

- `name`
- `package`
- `domain`

`project` optional fields:

- `tenancy`: `none`, `single`, `multi`
- `languages`: locale strings
- `currencies`: currency strings
- `frontend_framework`: `nextjs`, `remix`, `vite-react`
- `frontend.form_renderer`: `skeleton`, `real`
- `design.primary`: CSS hex
- `design.accent`: CSS hex
- `design.font_family`: CSS font family
- `design.density`: `compact`, `comfortable`
- `design.radius_scale`: float from `0.0` to `4.0`
- `compliance`
- `data_sensitivity`

`shared` optional fields:

- `roles`
- `permissions`
- `entities`

Every `jtbds[]` item requires:

- `id`
- `actor`
- `situation`
- `motivation`
- `outcome`
- `success_criteria` with at least one item

Common `jtbds[]` optional fields:

- `title`
- `version`
- `status`
- `edge_cases`
- `data_capture`
- `documents_required`
- `approvals`
- `sla`
- `notifications`
- `metrics`
- `requires`
- `compliance`
- `data_sensitivity`

Identifiers such as `project.package` and `jtbds[].id` must be ASCII
snake_case: start with a lowercase letter, then lowercase letters, digits, and
underscores.

Field kinds:

```text
text, number, money, date, datetime, enum, boolean, party_ref, document_ref,
email, phone, address, textarea, signature, file
```

The `jtbd-1.0` schema requires every data-capture field to declare
`pii: true` or `pii: false`; the parser additionally focuses errors for these
sensitive field kinds:

```text
email, phone, party_ref, signature, file, address, text, textarea
```

Edge-case handles:

```text
branch, reject, escalate, compensate, loop
```

`branch` requires `branch_to`.

Approval policies:

```text
1_of_1, 2_of_2, n_of_m, authority_tier
```

`n_of_m` requires `n`; `authority_tier` requires `tier`.

Notification triggers:

```text
state_enter, state_exit, sla_warn, sla_breach, approved, rejected, escalated
```

Notification channels:

```text
email, sms, slack, webhook, in_app
```

Compliance labels:

```text
GDPR, SOX, HIPAA, PCI-DSS, ISO27001, SOC2, NIST-800-53, CCPA
```

Sensitivity labels:

```text
PII, PHI, PCI, secrets, regulated
```

## Minimal JTBD Example

```yaml
project:
  name: claims-demo
  package: claims_demo
  domain: claims
  tenancy: single
jtbds:
  - id: claim_intake
    title: File a claim
    actor:
      role: claimant
      external: true
    situation: A claimant needs to report a loss.
    motivation: Start recovery quickly.
    outcome: A claim is ready for triage.
    success_criteria:
      - Claim ID is generated.
    data_capture:
      - id: claimant_name
        kind: text
        required: true
        pii: true
```

## Generation Workflow

1. Edit `jtbd-bundle.json` or `jtbd-bundle.yaml`.
2. Lint it:

```bash
uv run flowforge jtbd lint --bundle jtbd-bundle.json --warn-only
```

3. Generate:

```bash
uv run flowforge jtbd-generate --jtbd jtbd-bundle.json --out generated --force
```

4. Validate workflow definitions:

```bash
uv run flowforge validate --def generated/workflows/<jtbd_id>/definition.json
```

5. Simulate:

```bash
uv run flowforge simulate --def generated/workflows/<jtbd_id>/definition.json --events submit
```

6. Run tests appropriate to the change.

## Visual Workflow Editor

Package:

```text
js/flowforge-designer
```

Host dependency:

```json
{
  "dependencies": {
    "@flowforge/designer": "workspace:*"
  }
}
```

Basic React usage:

```tsx
import { Designer, sampleWorkflow } from "@flowforge/designer";

export function WorkflowEditorPage() {
  return (
    <main style={{ height: "100vh" }}>
      <Designer workflow={sampleWorkflow()} />
    </main>
  );
}
```

Store-backed usage:

```tsx
import { Designer, createDesignerStore } from "@flowforge/designer";

const store = createDesignerStore({ workflow, form });

export function WorkflowEditorPage() {
  return <Designer store={store} initialTab="canvas" />;
}
```

Tabs:

- `canvas`
- `form`
- `validation`
- `simulation`
- `diff`

Use `compareTo` for diff:

```tsx
<Designer workflow={draftWorkflow} compareTo={publishedWorkflow} initialTab="diff" />
```

In tests, pass `withReactFlow={false}` if the DOM is not measured:

```tsx
<Designer workflow={workflow} withReactFlow={false} />
```

Designer tests:

```bash
pnpm --dir js --filter @flowforge/designer test
pnpm --dir js --filter @flowforge/designer build
```

## Skinning

Generated host apps use `project.design`:

```yaml
project:
  design:
    primary: "#0f766e"
    accent: "#f59e0b"
    font_family: '"IBM Plex Sans", system-ui, sans-serif'
    density: compact
    radius_scale: 1.5
```

Regenerate after token changes. Host apps embedding the designer should map
these values to their own app shell and CSS variables.

## Code Map

- `python/flowforge-core`: DSL, compiler, engine, simulator, ports.
- `python/flowforge-cli`: CLI and JTBD generator command surfaces.
- `python/flowforge-jtbd`: canonical JTBD models and schema.
- `python/flowforge-fastapi`: HTTP/WS adapter.
- `python/flowforge-sqlalchemy`: durable Postgres storage.
- `js/flowforge-designer`: visual workflow editor.
- `js/flowforge-renderer`: form renderer and TS evaluator.
- `js/flowforge-runtime-client`: runtime REST/WS client.
- `examples/*/jtbd-bundle.json`: worked bundles.
- `docs/jtbd-user-guide.md`: human JTBD guide.
- `docs/jtbd-generation.md`: generator behavior.
- `docs/flowforge-handbook.md`: deeper architecture.

## Rules For Agents

- Prefer existing CLI commands and schemas over invented APIs.
- Do not add UMS as a dependency.
- Do not mutate generated output by hand when the source bundle should change.
- Regenerate after bundle/design-token changes.
- Validate and simulate workflow definitions after generation.
- Keep external release certification separate from ordinary package checks.
- Use in-memory port fakes for unit tests unless the task explicitly needs
  adapter or live Postgres behavior.
- For UI work, use the existing designer and generated design-token pipeline
  before adding new styling systems.
