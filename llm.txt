# Flowforge LLM Guide — v0.6.0 (2026-07-08)

Note: package metadata may still report `0.5.0` until the formal version bump;
this guide names the post-sprint hardening state as `v0.6.0`.

This file is the root entrypoint for AI agents working with Flowforge. Prefer
this file first, then [README.md](README.md), then
[docs/jtbd-user-guide.md](docs/jtbd-user-guide.md), then package-specific
READMEs.

## What Flowforge Is

Flowforge is a portable workflow framework for regulated, multi-tenant business
applications. It has:

- a JSON workflow DSL;
- a pure Python workflow engine;
- a strict expression evaluator for runtime/validator guard paths;
- host-wired ports for infrastructure;
- adapter packages for FastAPI, SQLAlchemy/Postgres, tenancy, audit, outbox,
  RBAC, documents, signing, notifications, and observability, all with real
  implementations in the current hardening state;
- TypeScript packages for rendering, runtime clients, a visual workflow
  designer, and JTBD editing;
- a deterministic JTBD-to-application generator.

UMS is not a Flowforge dependency. Use UMS only when an explicit downstream
release-certification task asks for UMS parity.

## Engine Capabilities (v0.6.0)

### Parallel Fork / Join

`fire()` accepts an optional `token_id` to advance a named branch token:

```python
from flowforge.engine import fire

# advance branch "branch_a" independently
await fire(wd, instance, event="approve", payload={}, token_id="branch_a")
# when all branch tokens drain, instance.state collapses to join state
```

Fork is enabled by default in v0.3.0+. To disable:

```bash
FLOWFORGE_FORKS_ENABLED=0 uv run flowforge ...
```

Or check at runtime:

```python
from flowforge.engine import forks_enabled
assert forks_enabled() is True  # default
```

### Fault Injection

```python
from flowforge.replay.fault import FaultInjector, FaultMode, FaultSpec

injector = FaultInjector([
    FaultSpec(mode=FaultMode.gate_fail, target_state="review")
])
result = await injector.simulate(wd, events=[("approve", {})])
```

CLI equivalent:

```bash
uv run flowforge simulate --def definition.json --fault gate_fail:review
```

### JWT / RBAC (hub)

```python
from flowforge_jtbd_hub.jwt_extractor import JwtPrincipalExtractor, make_jwt_extractor

extractor = make_jwt_extractor(signing)
token = await extractor.aissue_token(user_id, roles=["hub_admin"])
principal = await extractor(request)  # verifies + checks revocation
```

## Security Hardening (v0.6.0)

- Evaluator guard paths use `strict_ops=True`; unknown operators raise
  `EvaluationError` instead of returning truthy literal dictionaries.
- The FastAPI dashboard renders DB/query values through `_h()`, which wraps
  `html.escape(..., quote=True)`. Do not add dashboard HTML interpolation that
  bypasses `_h()`.
- JTBD pgvector store identifiers are regex-allowlisted through
  `_safe_identifier()` before SQL interpolation.
- `DocumentReviewStep` uses `sanitizeUrl()` and allows only `http:` and
  `https:` links. `javascript:` and `data:` URLs fall back to `#`.
- HMAC signature checks use timing-safe `hmac.compare_digest`.

## Main Commands

The root CLI is Rich-backed Typer. Global `--verbose` and `--quiet` are
available on the command surface.

```bash
uv run flowforge status [--json]
uv run flowforge tutorial --out /tmp/flowforge-demo --no-pause
uv run flowforge tutorial --domain insurance               # domain-specific walkthrough
uv run flowforge lint-jtbd examples/insurance_claim/jtbd-bundle.json
uv run flowforge new-workflow claim-intake --subject-kind claim
uv run flowforge jtbd lint --bundle examples/insurance_claim/jtbd-bundle.json --warn-only
uv run flowforge jtbd lint <bundle.json> [--strict]        # strict lifecycle lint
uv run flowforge jtbd lock --init <bundle.json>            # create lockfile
uv run flowforge jtbd lock --verify <bundle.json>          # verify lockfile matches
uv run flowforge jtbd fork <upstream> --tenant <id> [--out <path>]  # fork a JTBD bundle
uv run flowforge jtbd ai-draft "<description>" [--domain <d>]  # requires ANTHROPIC_API_KEY
uv run flowforge jtbd quality-score <bundle.json>          # score bundle quality
uv run flowforge jtbd compliance-lint <bundle.json> [--regime GDPR|HIPAA|SOX]
uv run flowforge jtbd-generate --jtbd examples/insurance_claim/jtbd-bundle.json --out /tmp/flowforge-insurance --force
uv run flowforge validate --def /tmp/flowforge-insurance/workflows/claim_intake/definition.json
uv run flowforge simulate --def /tmp/flowforge-insurance/workflows/claim_intake/definition.json --events submit
uv run flowforge simulate --def definition.json --fault gate_fail:review  # fault injection
```

Quality gates:

```bash
bash scripts/check_all.sh
make audit-2026-release-local
make audit-2026-release-external-preflight
```

The external release gate is manual release evidence. It needs browser support,
reviewed sidecar evidence, downstream UMS parity only when explicitly
configured, and live Postgres.

## Dashboard

`flowforge-fastapi` exposes `make_dashboard_router(session_factory, *,
prefix="/flowforge/dashboard", page_size=50, worker_pool=None)`.

The dashboard is server-rendered HTML with:

- dark-mode CSS custom properties;
- overview stat tiles;
- color-coded state/status badges;
- paginated instance and audit tables;
- responsive layout and accessible form labels;
- 30-second auto-refresh on the overview page.

Security rule: use existing dashboard helpers (`_h`, `_table`, `_badge`,
`_state_badge`, `_metric_card`, `_pagination`, `_json_block`, `_render_page`)
so all DB/query values are escaped before HTML rendering.

## JTBD Bundle Grammar

Bundles are JSON or YAML.

Top-level:

```yaml
project: {}
shared: {}
jtbds: []
```

`project` requires:

- `name`
- `package`
- `domain`

`project` optional fields:

- `tenancy`: `none`, `single`, `multi`
- `languages`: locale strings
- `currencies`: currency strings
- `frontend_framework`: `nextjs`, `remix`, `vite-react`
- `frontend.form_renderer`: `skeleton`, `real`
- `design.primary`: CSS hex
- `design.accent`: CSS hex
- `design.font_family`: CSS font family
- `design.density`: `compact`, `comfortable`
- `design.radius_scale`: float from `0.0` to `4.0`
- `compliance`
- `data_sensitivity`

`shared` optional fields:

- `roles`
- `permissions`
- `entities`

Every `jtbds[]` item requires:

- `id`
- `actor`
- `situation`
- `motivation`
- `outcome`
- `success_criteria` with at least one item

Common `jtbds[]` optional fields:

- `title`
- `version`
- `status`
- `edge_cases`
- `data_capture`
- `documents_required`
- `approvals`
- `sla`
- `notifications`
- `metrics`
- `requires`
- `compliance`
- `data_sensitivity`

Identifiers such as `project.package` and `jtbds[].id` must be ASCII
snake_case: start with a lowercase letter, then lowercase letters, digits, and
underscores.

Field kinds:

```text
text, number, money, date, datetime, enum, boolean, party_ref, document_ref,
email, phone, address, textarea, signature, file
```

The `jtbd-1.0` schema requires every data-capture field to declare
`pii: true` or `pii: false`; the parser additionally focuses errors for these
sensitive field kinds:

```text
email, phone, party_ref, signature, file, address, text, textarea
```

Edge-case handles:

```text
branch, reject, escalate, compensate, loop
```

`branch` requires `branch_to`.

Approval policies:

```text
1_of_1, 2_of_2, n_of_m, authority_tier
```

`n_of_m` requires `n`; `authority_tier` requires `tier`.

Notification triggers:

```text
state_enter, state_exit, sla_warn, sla_breach, approved, rejected, escalated
```

Notification channels:

```text
email, sms, slack, webhook, in_app, mail, push
```

Compliance labels:

```text
GDPR, SOX, HIPAA, PCI-DSS, ISO27001, SOC2, NIST-800-53, CCPA
```

Sensitivity labels:

```text
PII, PHI, PCI, secrets, regulated
```

## Minimal JTBD Example

```yaml
project:
  name: claims-demo
  package: claims_demo
  domain: claims
  tenancy: single
jtbds:
  - id: claim_intake
    title: File a claim
    actor:
      role: claimant
      external: true
    situation: A claimant needs to report a loss.
    motivation: Start recovery quickly.
    outcome: A claim is ready for triage.
    success_criteria:
      - Claim ID is generated.
    data_capture:
      - id: claimant_name
        kind: text
        required: true
        pii: true
```

## Generation Workflow

1. Edit `jtbd-bundle.json` or `jtbd-bundle.yaml`.
2. Lint it:

```bash
uv run flowforge jtbd lint --bundle jtbd-bundle.json --warn-only
```

3. Generate:

```bash
uv run flowforge jtbd-generate --jtbd jtbd-bundle.json --out generated --force
```

4. Validate workflow definitions:

```bash
uv run flowforge validate --def generated/workflows/<jtbd_id>/definition.json
```

5. Simulate:

```bash
uv run flowforge simulate --def generated/workflows/<jtbd_id>/definition.json --events submit
```

6. Run tests appropriate to the change.

## Visual Workflow Editor

Package:

```text
js/flowforge-designer
```

Host dependency:

```json
{
  "dependencies": {
    "@flowforge/designer": "workspace:*"
  }
}
```

Basic React usage:

```tsx
import { Designer, sampleWorkflow } from "@flowforge/designer";

export function WorkflowEditorPage() {
  return (
    <main style={{ height: "100vh" }}>
      <Designer workflow={sampleWorkflow()} />
    </main>
  );
}
```

Store-backed usage:

```tsx
import { Designer, createDesignerStore } from "@flowforge/designer";

const store = createDesignerStore({ workflow, form });

export function WorkflowEditorPage() {
  return <Designer store={store} initialTab="canvas" />;
}
```

Tabs:

- `canvas`
- `form`
- `validation`
- `simulation`
- `diff`

Current designer UX:

- draggable node palette for `state`, `transition`, `task`, and `gate`;
- position persistence;
- keyboard shortcuts: `Delete`, `Ctrl+Z`, `Ctrl+Y`, `Ctrl+A`, `E`, `Escape`;
- clickable validation errors;
- empty-canvas state.

Use `compareTo` for diff:

```tsx
<Designer workflow={draftWorkflow} compareTo={publishedWorkflow} initialTab="diff" />
```

In tests, pass `withReactFlow={false}` if the DOM is not measured:

```tsx
<Designer workflow={workflow} withReactFlow={false} />
```

Designer tests:

```bash
pnpm --dir js --filter @flowforge/designer test
pnpm --dir js --filter @flowforge/designer build
```

    accent: "#f59e0b"

`js/flowforge-jtbd-editor` is the bundle authoring surface. Current UX includes
a metadata editor panel, dependency editor popover, validation bottom bar, JSON
export, loading/error/empty states, and an O(1) edge map for dependency lookup.

## Rules For Agents

- Prefer existing CLI commands and schemas over invented APIs.
- Do not add UMS as a dependency.
- Do not mutate generated output by hand when the source bundle should change.
- Regenerate after bundle/design-token changes.
- Validate and simulate workflow definitions after generation.
- Keep external release certification separate from ordinary package checks.
- Use in-memory port fakes for unit tests unless the task explicitly needs
  adapter or live Postgres behavior.
- For UI work, use the existing designer and generated design-token pipeline
  before adding new styling systems.
- For dashboard work, escape all rendered DB/query values through `_h()`.
- For expression work, keep runtime/validator guard paths in strict operator
  mode.
